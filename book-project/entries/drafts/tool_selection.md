# tool selection

**Elevator definition**
Tool selection is how an agent chooses which tool to call for a given task — the decision that connects an agent's reasoning to its actions in the world.

## What it is

An agent without tools is a thinker. It can reason, plan, analyze, and generate text — but it cannot check a fact against a database, execute a calculation, send an email, or read a file. Tools are what give agents *agency*: the ability to act on the world rather than merely describe it. Tool selection is the decision that determines which action to take.

The mechanism works like this. An agent receives a task. It also receives a list of available tools, each with a name, a description, and a parameter schema. The agent reads the task, considers the tools, and decides: for this task, I need to call tool X with parameters Y. The decision is generated by the model as structured output (typically a JSON object with the tool name and arguments), which the runtime then executes. The tool returns a result, the agent reads the result, and the cycle continues.

This sounds straightforward. It is not. Tool selection is one of the hardest problems in agent design because it requires the model to make a *grounded decision* about the real world based on descriptions written in natural language. The model must understand what the tool does (from its description), whether the tool is appropriate for the current task (from reasoning about the task-tool fit), and how to parameterize the tool correctly (from the schema and the task requirements). Any of these can go wrong.

The quality of tool selection depends almost entirely on the quality of **tool descriptions**. A tool described as `search: searches things` tells the model almost nothing. A tool described as `search_knowledge_base: Searches the internal knowledge base for product documentation. Use when the user asks about product features, pricing, or compatibility. Input: a natural language query. Returns: up to 5 relevant document snippets with source URLs` tells the model exactly when to use it, what to expect, and what input to provide. The description is the tool's resume. If the resume is vague, the model will make bad hiring decisions.

Tool selection involves several distinct failure modes. **False selection** — the model picks a tool when no tool is needed. The user asks "What is 2 + 2?" and the agent calls a calculator API instead of just answering. **Missed selection** — the model should call a tool but doesn't, answering from its parametric knowledge when it should be checking a database. **Wrong selection** — the model picks the wrong tool, calling the email sender when it should have called the calendar checker. **Bad parameterization** — the model picks the right tool but provides incorrect parameters, searching for "revenue 2024" when the user asked about "revenue Q4 2025."

The decision space for tool selection grows combinatorially with the number of available tools. An agent with 3 tools can reasonably evaluate all options. An agent with 30 tools faces a selection problem that even frontier models struggle with. Research consistently shows that tool selection accuracy degrades as the number of available tools increases — models choose from a "favorite" subset and neglect tools they should use but rarely invoke.

Production systems manage this through **tool scoping**: not every tool is available to every agent on every turn. A routing layer or the orchestrator determines which subset of tools an agent can see based on the current task context. A billing agent sees billing tools. A research agent sees search tools. No agent sees all tools. This scoping reduces the decision space and improves selection accuracy.

## Why it matters in prompting

In prompted tool use (function calling), the tool descriptions you provide are part of the prompt. Poorly written descriptions waste tokens and confuse the model. Well-written descriptions are among the highest-ROI tokens in the entire prompt — they directly control whether the model uses tools correctly.

The pattern is the same as with any prompt instruction: specificity wins. "Use the search tool to find information" is weaker than "Use the search tool ONLY when the user asks about something that requires current information (events after January 2024, live pricing, stock levels). For general knowledge questions, answer directly without calling any tool." The second instruction prevents false selections while guiding correct ones.

Few-shot examples of correct tool use are disproportionately effective for tool selection. Showing the model one example of a query that should trigger a tool call, and one example of a similar query that should *not*, does more for selection accuracy than paragraphs of description.

## Why it matters in agentic workflows

In agent pipelines, tool selection is where the agent's reasoning meets reality. A planner agent that decides "I need to verify this claim" must select the right verification tool: is it a web search, a database query, a document lookup, or a calculation? The wrong selection produces garbage input for the next stage.

Tool selection also interacts with → routing. In some architectures, the routing decision and the tool selection are the same: the router is itself an agent that "selects" which downstream agent (treated as a tool) to invoke. In other architectures, routing happens first (assigning the task to an agent), and tool selection happens second (the assigned agent choosing which tools to use to complete its task). The distinction matters for system design: conflating the two layers makes debugging difficult.

## What it changes in model behavior

Tool availability fundamentally changes the model's generation strategy. A model without tools generates text to answer the question directly. A model with tools evaluates whether generating text or calling a tool is the better strategy for each sub-task. This evaluation is itself a form of reasoning — the model must assess its own knowledge boundaries and decide when to rely on itself versus when to consult an external source. Models with tool access are measurably more accurate on factual questions because they can verify rather than recall.

## Use it when

- The agent has access to external data sources, APIs, or computational tools
- The task requires information the model's training data may not contain (current data, private databases)
- Accuracy matters enough to justify the latency and cost of tool calls
- The agent must take actions (send emails, create records, trigger workflows) not just generate text
- Multiple tools are available and the correct choice depends on the input

## Do not use it when

- The task can be handled entirely by the model's built-in knowledge and reasoning
- The available tools do not match the task requirements (no tool solves the problem)
- Latency constraints prohibit the round-trip time of tool calls
- The model is used for pure generation (creative writing, brainstorming) where tools add nothing

## Contrast set

- → **routing** — Routing decides which *agent or path* handles a task. Tool selection decides which *tool within an agent* handles a sub-task. Routing is system-level dispatch. Tool selection is agent-level dispatch.
- → **delegation** — Delegation assigns work to another agent. Tool selection invokes a capability. Delegation is inter-agent; tool selection is agent-to-tool.
- → **orchestration** — Orchestration manages the overall pipeline flow. Tool selection is a local decision within a single agent's execution. Orchestration governs; tool selection acts.
- → **information routing** — Information routing decides what data goes where. Tool selection decides what action to take. Data flow versus action selection.

## Common failure modes

- **Description-driven hallucination** — The tool description is vague enough that the model infers capabilities the tool doesn't have. The model calls a "search" tool expecting it to search the internet, but the tool only searches a local database. Fix: be painfully explicit in descriptions. State what the tool does *and* what it does not do. Specify the scope of its data source.

- **Tool-happy agents** — The model calls tools for every query, including ones it could answer directly. Every question becomes a search query, even "What is the capital of France?" Fix: include explicit guidance on when *not* to call tools. "If the question can be answered from general knowledge, answer directly. Only call [tool] when the question requires [specific data type]."

- **Parameter hallucination** — The model constructs parameters that look plausible but don't match the tool's actual schema or the user's actual request. The user asks about "Q3 2025 revenue" and the model calls `get_revenue(year=2025, quarter=4)` — close but wrong. Fix: include parameter examples in the tool description. Validate parameters programmatically before execution. Return clear error messages when parameters are invalid, so the model can retry.

## Prompt examples

### Minimal example

```text
You have access to these tools:

1. search_docs(query: string) — Searches internal product
   documentation. Use for product feature or pricing questions.
2. check_order(order_id: string) — Looks up order status.
   Use when the user provides an order number.

If the user's question doesn't require either tool, answer
directly from your knowledge. Do not call a tool unless the
question specifically requires it.
```

### Strong example

```text
You are a customer support agent with the following tools.
Select a tool ONLY when the task requires it.

Tools:
1. search_kb(query: string)
   What it does: Searches the knowledge base for product
   documentation, policies, and how-to guides.
   Use when: User asks about product features, pricing,
   compatibility, setup instructions, or policies.
   Do NOT use when: User asks about their specific account,
   order, or personal information.
   Returns: Up to 5 document snippets with URLs.

2. lookup_account(email: string)
   What it does: Retrieves account details including plan,
   billing status, and recent activity.
   Use when: User asks about their account, billing, or
   subscription status.
   Requires: User's email address. If not provided, ask.
   Returns: Account summary JSON.

3. create_ticket(subject: string, priority: string, body: string)
   What it does: Creates a support ticket for human review.
   Use when: The issue cannot be resolved through knowledge
   base or account lookup, OR the user explicitly requests
   human assistance.
   Priority rules: "high" for billing errors, "medium" for
   feature issues, "low" for general feedback.

Decision rules:
- Try search_kb first for information questions.
- Try lookup_account for account-specific questions.
- Escalate to create_ticket only after attempting resolution.
- Never call a tool if you can answer confidently without one.
```

### Agentic workflow example

```text
Pipeline: Tool-Augmented Research Agent

Agent: Research Analyst
Available tools (scoped for this agent):

1. web_search(query: string, date_range: string | null)
   Scope: Public web. Use for current events, recent
   publications, market data after Jan 2024.

2. internal_search(query: string, corpus: string)
   Scope: Internal research corpus. Use for proprietary
   analysis, past research reports, internal data.
   Corpus options: "market_reports", "competitor_intel",
   "regulatory_filings"

3. calculate(expression: string)
   Scope: Mathematical computation. Use for growth rates,
   ratios, aggregations. Input: Python-style expression.

4. extract_table(document_id: string, table_index: int)
   Scope: Extracts structured data from documents in the
   corpus. Use when you need specific data from a known
   document.

Tool selection protocol:
- State your reasoning before each tool call: what you need,
  why this tool provides it, and what parameters are correct.
- If a tool returns an error, do not retry with the same
  parameters. Adjust your query or try a different tool.
- Maximum 5 tool calls per research task. Prioritize high-
  value calls. If you need more than 5, request escalation.
- After all tool calls, synthesize findings. Cite each
  tool result by its call ID in your synthesis.
```

## Model-fit note

Tool selection accuracy correlates strongly with model capability. Frontier models correctly select from 10+ tools with high accuracy when descriptions are clear. Midsize models handle 3–5 tools reliably but struggle with larger sets or subtle distinctions between similar tools. Small models are unreliable tool selectors and benefit from tool scoping (showing only 1–2 relevant tools per turn) or hard-coded routing that bypasses the model's selection entirely. For all tiers, tool descriptions are the highest-leverage investment — better descriptions improve selection more than model upgrades.

## Evidence and provenance

Tool use by language models is formalized in Schick et al. (2023) with Toolformer and extended by function calling capabilities in GPT-4 (2023) and Claude (2024). The degradation of tool selection accuracy with increasing tool count is documented in ToolBench evaluations. The Prompt Report discusses tool use as a prompting technique with specific recommendations for description quality [src_paper_schulhoff2025]. The concept of tool scoping for improving selection accuracy is a practitioner pattern observed in production agent systems.
